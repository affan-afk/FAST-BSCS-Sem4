Operating System Meaning

An operating system is a program that acts as an intermediary between a user and computer hardware. The slides define it as a resource allocator that manages CPU, memory, storage and I/O, and as a control program that prevents errors and improper use.

  • Kernel: the one program running at all times.
  • System program: shipped with the OS but not always running.
  • Application program: uses OS services to solve user problems.

Goals

Execute user programs, make problem solving easier, make the system convenient, and use hardware efficiently. Personal systems emphasize usability; shared systems emphasize fairness and utilization.

Computer System Structure

The four components are hardware, operating system, application programs and users. Hardware provides CPU, memory and I/O devices. The OS controls and coordinates use of hardware among applications and users.

Startup, Interrupts, Traps

A bootstrap program in ROM/EPROM firmware initializes the system, loads the kernel and starts execution. Device controllers inform the CPU of completion using interrupts. A trap or exception is a software-generated interrupt caused by error or user request. The OS saves CPU state, identifies the interrupt by polling or vectoring, runs the interrupt service routine, and resumes execution.

I/O and DMA

Device controllers have local buffers and device-status table entries. I/O may wait until completion or return control and later wait through a system call. Direct Memory Access lets high-speed devices transfer blocks directly to memory, causing one interrupt per block rather than per byte.

Storage, Caching, and Units

Main memory is directly accessible and volatile. Secondary storage is large and nonvolatile. Storage is organized by speed, cost and volatility. Caching copies active data from slow storage to faster storage, so cache size and replacement policy are important.

Architectures and Operations

Slides cover single-processor systems, special-purpose processors, multiprocessors, multicore systems and clustered systems. Multiprocessors improve throughput, economy of scale and reliability. Asymmetric multiprocessing assigns special tasks; symmetric multiprocessing lets all processors perform all tasks. OS operations include process, memory, storage, I/O, protection and security management.

Clustered Systems

Clustered systems are multiple systems working together, usually sharing storage via a storage-area network (SAN). They provide high-availability service that survives failures.

TypeDescription
Asymmetric clusteringOne machine in hot-standby mode monitors the active server. If it fails, the standby takes over.
Symmetric clusteringMultiple nodes run applications and monitor each other. More efficient use of hardware.
HPC clustersHigh-performance computing clusters run parallelized applications. Use distributed lock manager (DLM) to avoid conflicts.

Multiprogramming and Timesharing

Multiprogramming (batch system) organizes jobs so CPU always has one to execute. A subset of total jobs is kept in memory; one is selected via job scheduling. When a job waits (e.g., for I/O), the OS switches to another job. A single user cannot keep CPU and I/O busy at all times.

Timesharing (multitasking) is a logical extension where the CPU switches jobs so frequently that users can interact with each job while it runs, creating interactive computing. Response time should be less than 1 second.

  • If several jobs are ready: CPU scheduling decides which runs.
  • If processes don't fit in memory: swapping moves them in and out.
  • Virtual memory allows execution of processes not completely in memory.

Dual-Mode Operation

Dual-mode operation allows the OS to protect itself and other system components. The hardware provides a mode bit to distinguish between:

  • User mode (mode bit = 1): executing on behalf of a user application.
  • Kernel mode (mode bit = 0): executing on behalf of the operating system (also called supervisor, system, or privileged mode).

Some machine instructions are privileged and can only be executed in kernel mode (e.g., I/O control, timer management, interrupt management). A system call changes mode to kernel; return from call resets it to user.

Increasingly, CPUs support multi-mode operations, such as virtual machine manager (VMM) mode for guest VMs.

Timer

A timer prevents infinite loops and process resource hogging. The timer is set to interrupt the computer after a specified period.

  • A counter is decremented by the physical clock.
  • When counter reaches zero, an interrupt is generated.
  • The OS sets the counter (this is a privileged instruction).
  • The timer is set up before scheduling a process to regain control or terminate a program that exceeds allotted time.

Process Management

A process is a program in execution—a unit of work within the system. A program is a passive entity (on disk); a process is an active entity. A process needs resources: CPU, memory, I/O, files, and initialization data.

  • Single-threaded process has one program counter specifying the next instruction.
  • Multi-threaded process has one program counter per thread.
  • Concurrency is achieved by multiplexing the CPU among processes/threads.

OS Process Management Activities:

  • Creating and deleting user and system processes.
  • Suspending and resuming processes.
  • Providing mechanisms for process synchronization, communication, and deadlock handling.

Memory Management

To execute a program, all (or part) of its instructions and data must be in memory. Memory management determines what is in memory and when, optimizing CPU utilization and response time.

OS Memory Management Activities:

  • Keeping track of which parts of memory are currently being used and by whom.
  • Deciding which processes (or parts) and data to move into and out of memory.
  • Allocating and deallocating memory space as needed.

Storage (File-System) Management

The OS provides a uniform, logical view of information storage, abstracting physical properties to a logical storage unit called a file. Files are usually organized into directories. Most systems implement access control to determine who can access what.

OS Activities:

  • Creating and deleting files and directories.
  • Primitives to manipulate files and directories.
  • Mapping files onto secondary storage.
  • Backup files onto stable (non-volatile) storage media.

Mass-Storage Management

Disks store data that doesn't fit in main memory or must persist long-term. Proper management is critical—the entire speed of computer operation hinges on the disk subsystem.

OS Activities:

  • Free-space management.
  • Storage allocation.
  • Disk scheduling.

Tertiary storage includes optical storage and magnetic tape. Varies between WORM (write-once, read-many-times) and RW (read-write).

I/O Subsystem

One purpose of the OS is to hide peculiarities of hardware devices from the user. The I/O subsystem is responsible for:

  • Buffering: storing data temporarily while it is being transferred.
  • Caching: storing parts of data in faster storage for performance.
  • Spooling: overlapping output of one job with input of other jobs (e.g., print spooling).
  • General device-driver interface.
  • Drivers for specific hardware devices.

Data Migration and Cache Coherency

Data moves between levels of the storage hierarchy either explicitly or implicitly. In multitasking environments, the system must use the most recent value regardless of where it is in the hierarchy.

  • In multiprocessor environments, hardware must provide cache coherency so all CPUs have the most recent value in their cache.
  • In distributed environments, several copies of a datum can exist across multiple machines—even more complex solutions are required.

Protection and Security

Protection is any mechanism for controlling access of processes or users to resources defined by the OS. Security is defense of the system against internal and external attacks (denial-of-service, worms, viruses, identity theft).

  • User identities (user IDs, security IDs) include name and number, one per user.
  • User ID is associated with all files and processes of that user to determine access control.
  • Group identifier (group ID) allows sets of users to be defined and controls managed.
  • Privilege escalation allows a user to change to an effective ID with more rights.

Kernel Data Structures

The kernel uses many standard data structures internally:

Linked Lists

  • Singly linked list: each node points to the next.
  • Doubly linked list: each node has pointers to next and previous.
  • Circular linked list: the last node points back to the first.

Binary Search Tree

  • Left child ≤ right child at every node.
  • Search: O(n) worst case.
  • Balanced BST: O(log n) search.

Hash Maps

  • A hash function maps a key to a table index.
  • Used for fast lookup of kernel resources.

Bitmaps

  • A string of n binary digits representing the status of n items.
  • 0 = available, 1 = in use (or vice versa).

OS Services

The OS provides an execution environment and services to users, programs and other systems: user interface, program execution, I/O operations, file-system manipulation, communication, error detection, resource allocation, accounting, protection and security.

ServiceMeaning
Program executionLoad, run and terminate programs normally or abnormally.
I/O operationsRunning programs may require I/O involving files or devices.
File manipulationRead, write, create, delete, search, list and manage permissions.
CommunicationShared memory or message passing locally or over networks.
Error detectionOS constantly aware of errors in CPU, memory, I/O, or user programs.
Resource allocationAllocate CPU cycles, memory, file storage, I/O among concurrent users/jobs.
AccountingTrack which users use how much and what kinds of resources.
Protection/securityControlled resource access and defense from invalid access.

Interfaces: CLI, GUI, Touchscreen

CLI (command interpreter) allows direct command entry. Sometimes built into the kernel, sometimes a separate system program. Multiple flavors exist called shells. Commands may be built-in or names of programs—if the latter, adding features doesn't require shell modification.

GUI uses the desktop metaphor (invented at Xerox PARC). Icons represent files, programs, actions. Mouse buttons cause various actions. Examples: Windows GUI with CLI "command" shell, Mac OS X "Aqua" GUI with UNIX kernel, Linux with optional KDE/GNOME.

Touchscreen: mouse not possible—actions based on gestures, virtual keyboard, voice commands.

System Calls and APIs

System calls are the programming interface to OS services, typically written in C/C++. Programs use APIs rather than direct system calls: Win32 API, POSIX API, Java API.

The system-call interface maintains a table indexed by call numbers. It invokes the kernel routine and returns status/results. The caller needs to know nothing about implementation—details hidden by the run-time support library.

OS Design: Policy vs Mechanism

An important principle is to separate policy (what will be done?) from mechanism (how to do it?). This separation allows maximum flexibility—if policy decisions change later, only policies need updating, not mechanisms.

Implementation: Early OSes written in assembly. Now mostly C/C++ with lowest levels in assembly. Higher-level languages are easier to port but potentially slower. Emulation can allow an OS to run on non-native hardware.

  • Cocoa Touch: Objective-C API for apps.
  • Media services: graphics, audio, video.
  • Core services: cloud computing, databases.

  • Linux kernel: process, memory, device-driver management + power management.
  • Runtime: core libraries + Dalvik VM (Java bytecode → Dalvik executable).
  • Libraries: webkit (browser), SQLite (database), multimedia, smaller libc.
  • Apps developed in Java + Android API.

Debugging, Performance Tuning, DTrace

Debugging finds and fixes errors. OS generates log files for errors. Application failure can generate a core dump; OS failure can generate a crash dump.

Performance tuning removes bottlenecks. OS provides means like top (UNIX) or Task Manager (Windows). Profiling samples instruction pointer periodically for statistical trends.

DTrace (Solaris, FreeBSD, Mac OS X) allows live instrumentation on production systems. Probes fire when code executes within a provider, capturing state data and sending it to consumers.

Kernighan's Law: "Debugging is twice as hard as writing the code in the first place. Therefore, if you write the code as cleverly as possible, you are, by definition, not smart enough to debug it."

OS Generation (SYSGEN)

Operating systems are designed to run on any of a class of machines. SYSGEN program obtains information about the specific hardware configuration. Used to build a system-specific compiled kernel or system-tuned kernel. Can generate more efficient code than one general kernel.

System Boot

When power is initialized, execution starts at a fixed memory location. Firmware (ROM) holds initial boot code.

  • Small bootstrap loader (stored in ROM/EEPROM) locates the kernel, loads it into memory, starts it.
  • Sometimes a two-step process: boot block at fixed location loads bootstrap loader from disk.
  • Common bootstrap loader: GRUB—allows selection of kernel from multiple disks, versions, and kernel options.
Recall chain: application → API/runtime → system-call interface → kernel service → status/result.

Process Concept

A process is a program in execution. A program is passive on disk; a process is active in memory. Process memory contains text, data, heap and stack plus current activity through program counter and CPU registers.

  • New: being created.
  • Ready: waiting for CPU.
  • Running: executing instructions.
  • Waiting: waiting for I/O or event.
  • Terminated: finished execution.

PCB

The process control block stores process state, program counter, CPU registers, scheduling data, memory-management data, accounting and I/O status. Linux represents this through task_struct fields such as pid, state, time slice, parent, children, files and address space.

Process Scheduling and Context Switch

The scheduler selects a ready process for the CPU. Ready queues contain processes in memory waiting to execute; wait queues contain processes waiting for events. A context switch saves the old process state and loads the new process state; this time is pure overhead.

Creation and Termination

Parents create children, forming a process tree. fork() creates a new process, exec() replaces its memory with a new program, and wait() lets the parent collect termination status. A zombie has terminated but has not been waited for; an orphan's parent has terminated.

#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>

int main(void) {
 pid_t pid = fork(); // create a child process
 if (pid < 0) return 1;

 if (pid == 0) {
 printf("Child process\n");
 } else {
 wait(NULL); // parent collects child status
 printf("Parent process\n");
 }
 return 0;
}

The child receives return value 0 from fork(); the parent receives the child's pid and uses wait(NULL) to avoid a zombie.

POSIX Shared Memory

POSIX provides shared memory IPC. A process creates a shared memory segment, sets its size, and memory-maps it:

// Producer — POSIX Shared Memory
#include <fcntl.h>
#include <sys/mman.h>
#include <unistd.h>
#include <string.h>

int main() {
 const char *name = "/OS";
 const int SIZE = 4096;

 int shm_fd = shm_open(name, O_CREAT | O_RDWR, 0666);
 ftruncate(shm_fd, SIZE);

 void *ptr = mmap(0, SIZE, PROT_WRITE, MAP_SHARED, shm_fd, 0);
 sprintf(ptr, "Hello from producer!");

 return 0;
}

// Consumer — POSIX Shared Memory
#include <fcntl.h>
#include <sys/mman.h>
#include <stdio.h>

int main() {
 const char *name = "/OS";
 const int SIZE = 4096;

 int shm_fd = shm_open(name, O_RDONLY, 0666);
 void *ptr = mmap(0, SIZE, PROT_READ, MAP_SHARED, shm_fd, 0);

 printf("%s\n", (char *)ptr);
 shm_unlink(name);
 return 0;
}

shm_open creates/opens the segment, ftruncate sets its size, and mmap maps it into the process address space. Reading and writing use the pointer returned by mmap().

Producer-Consumer: Bounded Buffer (Shared Memory)

#define BUFFER_SIZE 10

typedef struct {
 int buffer[BUFFER_SIZE];
 int in;  // next free position
 int out; // first full position
} shared_data;

// Producer:
while (true) {
 while (((in + 1) % BUFFER_SIZE) == out)
  ; // buffer full, wait
 buffer[in] = next_produced;
 in = (in + 1) % BUFFER_SIZE;
}

// Consumer:
while (true) {
 while (in == out)
  ; // buffer empty, wait
 next_consumed = buffer[out];
 out = (out + 1) % BUFFER_SIZE;
}

This solution can hold at most BUFFER_SIZE - 1 items. The in pointer chases out in a circular fashion.

Thread Basics

A thread is a basic unit of CPU utilization. Threads in one process share code, data and files, but each has its own thread ID, program counter, registers and stack. Benefits include responsiveness, resource sharing, economy and scalability.

Concurrency vs Parallelism

Concurrency means multiple tasks make progress over time. Parallelism means tasks execute simultaneously on multiple cores. Multicore programming challenges include identifying tasks, balance, data splitting, data dependency, testing and debugging.

Speedup <= 1 / (S + (1 - S) / N)

User and Kernel Threads

User threads are managed by a user-level library; the kernel is unaware of them. Kernel threads are managed by the OS. Multithreading models include many-to-one, one-to-one, many-to-many and two-level models.

Thread Libraries

The slides include Pthreads, Windows threads and Java threads. Pthreads is the POSIX API for thread creation and synchronization.

#include <pthread.h>
#include <stdio.h>

void *worker(void *arg) {
 printf("Thread is running\n");
 return NULL;
}

int main(void) {
 pthread_t tid;
 pthread_create(&tid, NULL, worker, NULL);
 pthread_join(tid, NULL);
 return 0;
}

pthread_create starts a thread at the worker routine; pthread_join waits for it.

Implicit Threading and Issues

Implicit threading lets compilers/runtimes manage threads through thread pools, fork-join, OpenMP, Grand Central Dispatch and Intel TBB. Threading issues include fork/exec behavior, signal handling, thread cancellation, thread-local storage and scheduler activations. Linux uses clone() and refers to threads as tasks.

Amdahl's Law — Detailed

Identifies performance gains from adding cores to an application with both serial and parallel components.

Speedup ≤ 1 / (S + (1 - S) / N)

Where S = serial portion, N = number of processing cores.

ScenarioSerial (S)ParallelCores (N)Speedup
Example 125%75%21 / (0.25 + 0.75/2) = 1 / 0.625 = 1.6×
Example 225%75%41 / (0.25 + 0.75/4) = 1 / 0.4375 = 2.29×
Example 325%75%∞1 / (0.25 + 0) = 4× (max)
Example 450%50%∞1 / 0.50 = 2× (max)
Key insight: The serial portion has a disproportionate effect on performance. As N → ∞, speedup approaches 1/S. Even a small serial portion severely limits gains.

Data Parallelism

Distributes subsets of the same data across multiple cores, performing the same operation on each subset.

Example: Summing an array of N elements. Thread 1 sums elements [0, N/2-1], Thread 2 sums elements [N/2, N-1]. Results are combined.

Task Parallelism

Distributes threads (not data) across cores, each thread performing a unique operation.

Example: In a web server, one thread handles network I/O while another processes business logic and a third manages database queries.

Thread Pools

Create a pool of threads at process startup and assign tasks from a queue. Benefits:

  • Servicing a request with an existing thread is faster than creating a new thread.
  • A thread pool limits the number of threads that exist at any one point.
  • Separating task creation from task execution allows different strategies for scheduling.

Signal Handling in Threads

Signals notify a process that a particular event has occurred. A signal handler processes signals:

  1. Signal is generated by a particular event.
  2. Signal is delivered to a process.
  3. Signal is handled by default or user-defined handler.

In multithreaded programs, where should the signal be delivered?

  • Deliver to the thread to which the signal applies.
  • Deliver to every thread in the process.
  • Deliver to certain threads only.
  • Assign a specific thread to receive all signals.

Thread Cancellation

Terminating a thread before it has finished. The target thread can be cancelled in two ways:

TypeBehavior
AsynchronousTerminates the target thread immediately. Dangerous—may leave shared data in inconsistent state.
DeferredTarget thread periodically checks if it should be cancelled at cancellation points. Allows cleanup.
pthread_t tid;
pthread_create(&tid, NULL, worker, NULL);
// ...
pthread_cancel(tid); // request cancellation

Thread-Local Storage (TLS)

TLS allows each thread to have its own copy of data. Useful when you don't control thread creation (e.g., thread pools).

ConceptScope
Local variablesVisible only during a single function invocation.
TLSVisible across function invocations, but unique to each thread.
Static dataShared across all threads (global).

Linux Threads and Scheduler Activations

Linux refers to threads as tasks. Thread creation is done through clone() system call, which allows a child task to share the address space of the parent. Flags control behavior (share files, memory, signals, etc.).

Scheduler activations provide upcalls—a communication mechanism from the kernel to the user-level thread library. The kernel provides a virtual processor (LWP — Lightweight Process) to each application, and user threads are scheduled onto available LWPs.

Basic Concepts

Maximum CPU utilization is obtained with multiprogramming. Process execution alternates between CPU bursts and I/O bursts. Scheduling chooses the next ready process; the dispatcher switches context, switches to user mode and jumps to the correct program location. Dispatch latency is the time needed to stop one process and start another.

Preemptive vs Nonpreemptive

Nonpreemptive scheduling lets a process keep the CPU until it blocks or finishes. Preemptive scheduling can take the CPU away when a better candidate appears or a time quantum expires.

FCFS

First-Come, First-Served is simple and nonpreemptive. It may cause the convoy effect when short jobs wait behind a long CPU-bound job.

SJF and SRTF

Shortest-Job-First chooses the shortest next CPU burst and is optimal for minimum average waiting time for a fixed set. The hard part is predicting the next burst. Shortest-Remaining-Time-First is the preemptive version.

Race Condition

A race condition occurs when several processes access and manipulate shared data concurrently and the result depends on the order of access. The slide example uses a kernel variable such as next_available_pid.

Critical Section Requirements

  • Mutual exclusion: only one process in the critical section.
  • Progress: if no one is inside, selection is not postponed indefinitely.
  • Bounded waiting: no process waits forever.

Peterson Solution

boolean flag[2] = {false, false};
int turn;

// process i, other process j = 1 - i
flag[i] = true;
turn = j;
while (flag[j] && turn == j) {
 ; // busy wait
}
/* critical section */
flag[i] = false;

flag announces interest; turn gives priority to the other process. The loop waits only while the other process wants entry and it is the other's turn. Modern CPUs may reorder memory operations, so Peterson is not reliable without proper memory-ordering support.

Mutex Locks

A mutex protects critical sections with acquire/release behavior. Busy waiting is acceptable only for very short waits; blocking waits avoid wasting CPU.

pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;
int counter = 0;

void *increment(void *arg) {
 for (int i = 0; i < 100000; i++) {
 pthread_mutex_lock(&lock); // entry section
 counter++; // critical section
 pthread_mutex_unlock(&lock); // exit section
 }
 return NULL;
}

Semaphores

A semaphore is an integer synchronization tool accessed by atomic wait() and signal(). Counting semaphores have unrestricted range; binary semaphores range 0 to 1 and behave like mutexes.

sem_t mutex;
sem_init(&mutex, 0, 1);

sem_wait(&mutex); // acquire / P operation
/* critical section */
sem_post(&mutex); // release / V operation

sem_destroy(&mutex);

No Busy Waiting and Liveness

Semaphore implementations can block processes in a waiting queue and wake them with signal(). Liveness means desired events eventually occur. Failures include deadlock, starvation and livelock.

Critical Section Problem Structure

Each process follows this code pattern:

do {
 [entry section]
  critical section
 [exit section]
  remainder section
} while (true);

The entry section requests permission to enter the critical section. The exit section signals departure. The remainder section is the rest of the code.

Interrupt-Based Solution

Disable interrupts in the entry section, enable in the exit section. Problems:

  • If the critical section runs for a long time, other processes starve.
  • Does not work on multiprocessor systems—disabling interrupts on one CPU doesn't affect others.
  • Generally too inefficient for user-level synchronization.

Memory Barrier Instructions

Modern CPUs reorder memory operations for performance. This can break Peterson's solution on real hardware. Memory barriers (fences) ensure ordering:

// Without barriers — may output 0!
Thread 1: while (!flag) ; print x
Thread 2: x = 100; flag = true;

// With barriers — guaranteed to output 100
Thread 1: while (!flag) memory_barrier(); print x
Thread 2: x = 100; memory_barrier(); flag = true;

The memory barrier in Thread 2 ensures x = 100 completes before flag = true. In Thread 1, the barrier ensures flag is loaded before x.

Hardware Atomic Instructions

Many systems provide hardware atomic instructions that execute as indivisible units:

test_and_set()

boolean test_and_set(boolean *target) {
 boolean rv = *target;
 *target = true;
 return rv;
}

// Usage:
while (test_and_set(&lock))
 ; // busy wait
// critical section
lock = false;

compare_and_swap()

int compare_and_swap(int *value,
                     int expected,
                     int new_val) {
 int temp = *value;
 if (*value == expected)
  *value = new_val;
 return temp;
}

// Usage:
while (compare_and_swap(&lock, 0, 1) != 0)
 ; // busy wait
// critical section

Semaphore Implementation with No Busy Waiting

Each semaphore has an associated waiting queue. Two operations:

  • block(): place the invoking process on the waiting queue.
  • wakeup(): remove a process from the waiting queue and place it in the ready queue.
typedef struct {
 int value;
 struct process *list;
} semaphore;

wait(semaphore *S) {
 S->value--;
 if (S->value < 0) {
  add this process to S->list;
  block();
 }
}

signal(semaphore *S) {
 S->value++;
 if (S->value <= 0) {
  remove a process P from S->list;
  wakeup(P);
 }
}

Note: when S->value < 0, its absolute value equals the number of processes waiting on that semaphore.

Peterson’s solution is not guaranteed to work on modern computer architectures for the primary reason that, to improve system performance, processors and/or compilers may reorder read and write operations that have no dependencies. Monitor Solution: Dining Philosophers

Each philosopher is THINKING, HUNGRY, or EATING. A philosopher can eat only if neither neighbor is eating.

monitor DiningPhilosophers {
 enum {THINKING, HUNGRY, EATING} state[5];
 condition self[5];

 void pickup(int i) {
  state[i] = HUNGRY;
  test(i);
  if (state[i] != EATING) self[i].wait();
 }

 void putdown(int i) {
  state[i] = THINKING;
  test((i + 4) % 5); // test left neighbor
  test((i + 1) % 5); // test right neighbor
 }

 void test(int i) {
  if (state[(i+4)%5] != EATING &&
      state[i] == HUNGRY &&
      state[(i+1)%5] != EATING) {
   state[i] = EATING;
   self[i].signal();
  }
 }
}

  in = (in + 1) % SIZE;
  count++;

  notify();

Definition

A set of processes is deadlocked when each process is blocked waiting for an event that only another blocked process in the same set can trigger. Deadlock is permanent unless recovery action occurs.

Traffic Example

The slides distinguish potential deadlock from actual deadlock using cars at an intersection. Actual deadlock occurs when each car seizes one quadrant and waits for another already seized quadrant.

Four Conditions

ConditionMeaning
Mutual exclusionOnly one process may use a nonsharable resource at a time.
Hold and waitA process holds resources while waiting for others.
No preemptionResources cannot be forcibly removed.
Circular waitA closed chain exists where each process waits for the next.

Resource-Allocation Graphs

A process-to-resource edge is a request. A resource-instance-to-process edge is an assignment. With one instance per resource, a cycle implies deadlock; with multiple instances, a cycle indicates possible deadlock.

Prevention, Avoidance, Detection

Prevention breaks one necessary condition by design. Avoidance dynamically decides whether a request may lead to deadlock. Detection allows deadlock, detects it and recovers.

Safe State

A safe state has at least one safe sequence in which every process can finish. Unsafe means deadlock is possible, not guaranteed.

Banker's Algorithm

Data structures: Available, Max, Allocation, and Need = Max - Allocation. Grant a request only if pretending to allocate it leaves the system safe.

Detection and Recovery

Detection uses wait-for graphs for single-instance resources or matrix detection for multiple instances. Recovery may terminate processes or preempt resources; preemption must choose a victim, roll back and avoid starvation.

Safety Algorithm — Step by Step

  1. Let Work = Available and Finish[i] = false for all i.
  2. Find process i where Finish[i] = false and Need[i] ≤ Work. If none exists, go to step 4.
  3. Work = Work + Allocation[i]; Finish[i] = true. Go to step 2.
  4. If Finish[i] = true for all i → system is in safe state.

Resource-Request Algorithm

When process Pi requests resources Requesti:

  1. If Requesti ≤ Needi, go to step 2. Otherwise, error (exceeded max claim).
  2. If Requesti ≤ Available, go to step 3. Otherwise, Pi must wait.
  3. Pretend to allocate: Available -= Requesti; Allocationi += Requesti; Needi -= Requesti.
  4. Run Safety Algorithm. If safe → allocate. If unsafe → Pi waits, restore old state.

Recovery from Deadlock — Detailed

Process Termination

  • Abort all deadlocked processes (expensive).
  • Abort one at a time until cycle is broken. Selection criteria:
  • Priority of the process
  • How much computation done/remaining
  • Resources held and resources needed
  • How many processes need terminating

  • Rollback: return to some safe state, restart process.
  • Starvation prevention: include number of rollbacks in cost factor so same process isn't always picked as victim.

Background and Protection

Main memory and registers are directly accessible to the CPU. Processes must be loaded into memory to run. Protection can use base and limit registers so every logical address is checked before physical access.

Address Binding

Binding can occur at compile time, load time or execution time. Execution-time binding uses hardware and allows logical and physical addresses to differ while the process runs.

Logical, Physical, MMU

A logical address is generated by the CPU and is also called a virtual address. A physical address is seen by memory. The Memory-Management Unit maps virtual to physical addresses at runtime.

physical address = relocation register + logical address

Dynamic Loading and Linking

Dynamic loading brings a routine into memory only when called. Dynamic linking postpones linking until execution and is useful for shared libraries.

Contiguous Allocation

Each process occupies one contiguous memory region. Dynamic allocation chooses holes using first fit, best fit or worst fit.

Fragmentation

External fragmentation means total free space is enough but not contiguous. Internal fragmentation means allocated memory is larger than requested. Compaction reduces external fragmentation only when relocation is dynamic.

Worst internal fragmentation = 1 frame - 1 byte; average = 1/2 frame.

Paging

Paging divides physical memory into frames and logical memory into equal-size pages. The page table translates page number to frame address. Paging avoids external fragmentation but still has internal fragmentation.

For address space 2^m and page size 2^n: page number bits = m-n, offset bits = n.

TLB and Page Tables

A TLB is a fast associative cache of page-table entries. Hit ratio is the percentage of page numbers found in the TLB. Large page tables motivate hierarchical paging, hashed tables and inverted page tables.

Swapping and Architectures

Swapping moves processes between memory and backing store; roll out/roll in supports priority scheduling. Slides include IA-32 segmentation with paging, PAE, x86-64 four-level paging and ARM architecture with TLB levels.

Hardware Address Protection

The CPU must check every memory access against base and limit registers. If the logical address is ≥ limit, a trap to the OS occurs. Otherwise, the base register value is added.

Multistep Processing of a User Program

A user program goes through several steps before execution:

  1. Source code → Compiler → Object module
  2. Object module + other modules → Linker → Load module
  3. Load module → Loader → In-memory binary image

Addresses can be bound at compile time (absolute), load time (relocatable), or execution time (using MMU relocation register).

Variable Partition Allocation

In variable-partition schemes, the OS maintains information about allocated and free partitions (holes).

StrategyRulePros/Cons
First FitAllocate first hole big enough.Fast. Good enough in practice.
Best FitAllocate smallest hole that's big enough.Leaves smallest leftover hole. Must search entire list.
Worst FitAllocate largest hole.Leaves largest leftover. Must search. Generally worst performance.

50-percent rule: Statistical analysis shows that given N blocks allocated, approximately 0.5N blocks are lost to fragmentation—1/3 of memory may be unusable.

Compaction

Reduce external fragmentation by shuffling memory contents to place all free memory in one large block. Only possible if relocation is dynamic (at execution time).

I/O problem: Cannot compact if a process has pending I/O to its memory region. Solutions: latch job in memory during I/O, or use double buffering (always transfer I/O to OS buffer first).

Paging Example — Numerical

Given: n = 2 (page offset bits), m = 4 (logical address bits), page size = 4 bytes, physical memory = 32 bytes (8 frames).

  • Page number bits = m - n = 4 - 2 = 2 bits (4 pages)
  • Page offset bits = n = 2 bits
  • Physical address = frame number (3 bits) + offset (2 bits) = 5 bits

Logical address 0 → page 0, offset 0 → frame 5 → physical 10100 = 20
Logical address 3 → page 0, offset 3 → frame 5 → physical 10111 = 23
Logical address 4 → page 1, offset 0 → frame 6 → physical 11000 = 24

Internal Fragmentation Calculation

Given: Page size = 2,048 bytes, Process size = 72,766 bytes.

Pages needed = ⌈72,766 / 2,048⌉ = 35 pages + 1,086 bytes remainder.
Memory allocated = 36 × 2,048 = 73,728 bytes.
Internal fragmentation = 2,048 - 1,086 = 962 bytes.
Worst case = 1 frame - 1 byte. Average = 1/2 frame size.

Page Table Implementation

  • Page table kept in main memory.
  • PTBR (Page-Table Base Register) points to the page table.
  • PTLR (Page-Table Length Register) indicates size of page table.
  • Every data/instruction access requires two memory accesses: one for the page table, one for the data.
  • The TLB (Translation Look-Aside Buffer) solves this two-access problem.

TLB — Detailed

A fast associative cache of page-table entries, typically 64 to 1,024 entries.

  • ASID (Address-Space Identifier): stored in each TLB entry; uniquely identifies each process. Without ASIDs, the TLB must be flushed on every context switch.
  • Wired entries: cannot be evicted. Used for critical kernel pages.
  • Replacement policy: LRU is common but hardware-specific.

Memory Protection and Valid-Invalid Bit

Each page table entry has a valid-invalid bit:

  • Valid (v): the page is in the process's logical address space and is mapped to a physical frame.
  • Invalid (i): the page is NOT in the logical address space (or hasn't been loaded yet).

Any access to an invalid page causes a trap to the OS. Additional protection bits indicate read-only, read-write, or execute-only.

Shared Pages

Reentrant code (read-only, pure code) can be shared among processes. Each process has its own page table entry pointing to the same physical frame.

Example: 40 users running a text editor (150KB code + 50KB data): without sharing = 40 × 200KB = 8,000KB; with sharing = 150KB + 40 × 50KB = 2,150KB. Savings = 5,850KB.

Virtual Memory

Virtual memory separates user logical memory from physical memory. It lets programs be larger than RAM, supports sparse address spaces, shared libraries, shared memory and efficient process creation.

Demand Paging

Demand paging loads pages only when needed. The valid-invalid bit marks whether a page is resident/legal. An invalid resident reference causes a page fault.

Page Fault Handling

  1. Trap to OS.
  2. Check whether reference is legal.
  3. Find a free frame or select a victim.
  4. Read required page from disk.
  5. Update page/frame tables and valid bit.
  6. Restart the faulting instruction.

Performance

EAT = (1 - p) * ma + p * page_fault_time

p is page-fault rate. The slide example with 200 ns memory access, 8 ms fault service time and p=0.001 gives about 8.199 microseconds, showing page faults dominate performance.

Copy-on-Write

COW lets parent and child initially share pages after fork(). Only modified pages are copied, so process creation is faster.

Page Replacement

When no frame is free, choose a victim page. Dirty victims must be written to disk before reuse. Page and frame tables are updated afterward.

Algorithms

AlgorithmRuleExam note
FIFOReplace oldest loaded page.Simple; can suffer Belady anomaly.
OptimalReplace page used farthest in future.Best theoretical benchmark.
LRUReplace page unused for longest past time.Practical approximation of optimal.
LRU approximationsUse reference bits/clock behavior.Avoid exact LRU overhead.

Frame Allocation and Thrashing

Equal and proportional allocation decide frame counts. Global replacement can take frames from any process; local replacement uses only a process's own frames. Thrashing occurs when a process spends most time paging rather than executing.

Kernel Memory

Kernel memory may need contiguous physical pages. Buddy allocation uses power-of-two blocks. Slab allocation caches initialized kernel objects in slabs.

Prepaging and I/O Interlock

Prepaging loads likely-needed pages early but wastes I/O and memory if unused. I/O interlock locks pages involved in device I/O so replacement cannot evict them mid-transfer.

Demand Paging Optimizations

Demand Paging EAT — Detailed Calculation

EAT = (1 - p) × ma + p × page_fault_time

Given: Memory access time (ma) = 200 ns, average page-fault service time = 8 ms, p = 1/1000 = 0.001

EAT = (1 - 0.001) × 200 + 0.001 × 8,000,000
= 0.999 × 200 + 8,000
= 199.8 + 8,000
= 8,199.8 ns ≈ 8.2 μs

This represents a 40× slowdown compared to 200 ns! Even a tiny page fault rate dominates performance.

Basic Page Replacement Steps

  1. Find the location of the desired page on disk.
  2. Find a free frame: if no free frame, use page replacement algorithm to select a victim frame.
  3. If victim is dirty (modify bit = 1), write it to disk first.
  4. Bring the desired page into the newly free frame.
  5. Update page and frame tables.
  6. Continue the process by restarting the faulting instruction.

Note: A dirty page replacement requires two disk transfers (write victim + read new page), doubling the page fault time. A modify (dirty) bit reduces transfers by skipping writes for unmodified pages.

Belady's Anomaly

With FIFO, increasing the number of frames can increase page faults. This counter-intuitive behavior is called Belady's anomaly. Example: reference string 1,2,3,4,1,2,5,1,2,3,4,5 with 3 frames gives 9 faults, but with 4 frames gives 10 faults!

LRU Approximation — Second-Chance (Clock) Algorithm

Uses a reference bit set by hardware on each access:

  1. Arrange frames in a circular queue with a clock pointer.
  2. When replacement needed, examine page at clock pointer:
  3. If reference bit = 0: replace this page.
  4. If reference bit = 1: set bit to 0, advance pointer, check next page.

Enhanced version uses both reference bit (R) and modify bit (M): (0,0) best victim → (0,1) → (1,0) → (1,1) worst victim. May need to scan circular queue multiple times.

Thrashing — Working Set Model

Thrashing = process spends more time paging than executing. Cause: total memory demanded by all processes exceeds physical memory.

Working-Set Model: Define Δ (working-set window). WSi = set of pages referenced in the most recent Δ references. If Σ WSi > available frames, thrashing occurs → suspend a process.

Buddy System — Numerical Example

Given: 256 KB chunk available, kernel requests 21 KB.

Step 1: 256KB → split into 128KB + 128KB (AL and AR).
Step 2: 128KB (AL) → split into 64KB + 64KB (BL and BR).
Step 3: 64KB (BL) → split into 32KB + 32KB (CL and CR).
Step 4: 32KB ≥ 21KB → allocate CL (32KB) to satisfy the 21KB request.
Internal fragmentation: 32 - 21 = 11 KB wasted.
On deallocation, buddies are coalesced back into larger blocks.

Slab Allocator — Detailed

A slab = one or more physically contiguous pages. A cache = one or more slabs for a specific kernel data structure.

  • When cache is created, objects are allocated and marked free.
  • When kernel needs an object, one is taken from cache and marked used.
  • If slab is full, next object comes from an empty slab.
  • If no empty slabs, a new slab is allocated.

Benefits: No fragmentation (objects are exact size). Fast allocation (pre-initialized objects). Linux uses SLUB allocator (simplified slab).

Memory-Mapped Files

Memory-mapped file I/O maps a disk block to a page in memory. File I/O is treated as routine memory access. Multiple processes can map the same file, enabling shared memory. Writes to the mapped region are written back to disk on munmap() or periodically.

Slide 28: Optimal Page Replacement Algorithm

To Overcome Belady’s anomaly, optimal page-replacement algorithm introduced—the algorithm that has the lowest page-fault rate of all algorithms and will never suffer from Belady’s anomaly. Such an algorithm does exist and has been called OPT or MIN. It is simply this: Replace the page that will not be used for the longest period of time. Use of this page-replacement algorithm guarantees the lowest possible page- fault rate for a fixed number of frames.

Slide 37: Fixed Allocation

Equal allocation – For example, if there are 100 frames (after allocating frames for the OS) and 5 processes, give each process 20 frames Keep some as free frame buffer pool Proportional allocation – Allocate according to the size of process Dynamic as degree of multiprogramming, process sizes change

Slide 38: Priority Allocation

Use a proportional allocation scheme using priorities rather than size If process Pi generates a page fault, select for replacement one of its frames select for replacement a frame from a process with lower priority number

Slide 39: Global vs. Local Allocation

Global replacement – process selects a replacement frame from the set of all frames; one process can take a frame from another But then process execution time can vary greatly But greater throughput so more common Local replacement – each process selects from only its own set of allocated frames More consistent per-process performance But possibly underutilized memory

Slide 40: Thrashing

If a process does not have “enough” pages, the page-fault rate is very high Page fault to get page Replace existing frame But quickly need replaced frame back This leads to: Low CPU utilization Operating system thinking that it needs to increase the degree of multiprogramming Another process added to the system Thrashing ≡ a process is busy swapping pages in and out

Slide 43: Buddy System

Allocates memory from fixed-size segment consisting of physically-contiguous pages Memory allocated using power-of-2 allocator Satisfies requests in units sized as power of 2 Request rounded up to next highest power of 2 When smaller allocation needed than is available, current chunk split into two buddies of next-lower power of 2 Continue until appropriate sized chunk available For example, assume 256KB chunk available, kernel requests 21KB Split into AL and AR of 128KB each One further divided into BL and BR of 64KB One further into CL and CR of 32KB each – one used to satisfy request Advantage – quickly coalesce unused chunks into larger chunk Disadvantage - fragmentation

Slide 45: Slab Allocator

Alternate strategy Slab is one or more physically contiguous pages Cache consists of one or more slabs Single cache for each unique kernel data structure Each cache filled with objects – instantiations of the data structure When cache created, filled with objects marked as free When structures stored, objects marked as used If slab is full of used objects, next object allocated from empty slab If no empty slabs, new slab allocated Benefits include no fragmentation, fast memory request satisfaction
